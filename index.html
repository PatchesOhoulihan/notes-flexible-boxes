Responsive Web Design unterstützt die flexible Natur des Web

Webweisender Artikel von Ethan Marcotte check here

Marcotte beschreibt in seinem Artikel drei Techniken, mit denen man seiner Meinung nach mediengerechte Webseiten bauen sollte.

  1. prozentbasierte, flexible Gridlayouts(Fluidgrids)

  2. flexible Bilder

  3. Media Queries

Vereint ermöglichen diese drei Techniken, dass eine Website auf die Umgebung reagiert und sich z.B der Größe des Browserfensters, oder genau genommen der Größe des Viewports, anpasst.

Die grundidee des responsive Webdesign ist es, ein und denselben HTML-Quelltext per Media Query mit unterschiedlichen CSS auszuliefern, sodass sie sich den unterschiedlichsten Umgebungen anpassen.

Responsive vs. Adaptive Layouts

Marcotte bevorzugt in seinem Artikel ein flexibles prozentbasiertes Raster. Aber nicht alle benutzten dieses, und so bürgerten sich im allgemeinen Sprachgebrauch die Unterscheidung zwischen responsive und adaptiven Layouts

  • Ein responsive Layout ist stufenlos und enthält alle drei von Marcotte propagierten Komponenten, also prozentbasiertes Grid, flexible Medien und Media Queries.

  • Ein adaptives Layout verzichtet auf das schwieriger umzusetzende Fluid Grid und liefert stattdessen per Media Query unterschiedliche feste Layoutbreiten aus.

Natürlich wurden darauf hin diskussionen geführt was jetzt responsiv ist.

Daraufhin meldete sich Jeffrey Zeldmann zu Wort, der die Meinung vertrat das responsive Webdesign nicht an eine bestimmte Technik gekoppelt werden sollte. siehe hier.

Die Quintessenz ist, dass es egal ist ob man es nun responsive oder adaptive nennt, gemeint ist letztlich alles, was Webseiten die Flexibilität gibt, um auf unterschiedlichsten Geräten eien gute Figur zu machen.

Moderne Webseiten erstellen

Bei der Erstellung des HTML, CSS und Javascript für moderne Webseiten gibt es zwei grundsätzliche verschiedene Herangehensweisen:

  1. Desktop first beginnen mit dem komplexesten Layout, zuweist auf Computer Monitor, und reduziert dieses dann für kleinere Bildschirme.

  2. Mobile first beginnt mit der kleinsten Variante und fügt für größere Bildschirme nach und nach Dinge hinzu.

Das wichtigste verfahren ist jedoch Content first. Wirklich wichtig ist der Inhalt. Gestaltung ohne Inhalt ist nicht mehr als eine Dekoration. Der Content der Seite ist der Grund warum jemand diese besucht. Jeffrey Zeldman hat diese simple Tatsache so ausgedrückt: Websites are simply delivery systems for content.

Semantische Strukturelemente in HTML5

In HTML5 gibt es neue Elemente, mit denen man die Struktur einer Seite auszeichnen kann. Gemeint sind damit Dinge wie Kopfbereich, Navigation, Inhaltsbereich, Sidebar und Fußbereich.

Übersicht Semantische Elemente HTML 5
HTML-Element Kurzbeschreibung
header Kopfbereiche, zumeist direkt hinter dem body-tag zumeist für logos, slogan und/oder Navigation(auch das semantische Nav). Kann mehrfach auf der Seite vorkommen.
nav Navigationsblöcke, diese tag ist für die wichtigsten Navigationsblöcke der Seite reserviert und nicht für eine einfache link sammelung verwendet werden.
main Hauptinhalt einer Seite, markiert den Hauptinhalt einer Seite, darf auf jeder Seite nur ein einziges mal vorkommen, und darf nicht in einem anderen Semantischen Tag stehen(nav, footer, aside)
section Einteilung in inhaltliche Abschnitte, geht es nur um die Gruppierung von Elementen nur um die Gestalutng, ist ein div eine bessere Wahl. Bespielweise lässt sich das main mit mehreren sections unterteilen, wie getting started und darauf further reading. Können ineinander verschachtelt werden

aside zusätzliche Informationen, die den umgebenden Text erläutern oder nur marginal mit ihm in Verbindung stehen. Nimmt man diesen Text aus dem Dokument heraus, sollte der Haupt Inhalt immer noch verständlich sein. Z.B Werbung.
footer Fußbereich, befinden sich zumeist ergänzende Informationen. Ist nicht nur für die ganze Seite erlaubt, sondern kann auch in Sections und Artikeln verwendet werden. Der Footer bezieht sich immer auf das Element in dem er sich befindet.

Article vs Section

Für die richtige verwendung sollte sich folgendes Beispiel vor augen gehalten werden.

  • Eine Zeitung besteht aus Abschnitten für Nachrichten, Sport und Börse etc. Jeder dieser Abschnitte ist eine section

  • Innerhalb eines Abschnitt gibt es Artikel. Ein Artikel ist eine in sich geschlossene Komposition.

Semantisches HTML gibt den Inhalten eine Bedeutung.

Denn noch lassen sich verschiedene Elemente durch CSS gleich aufstylen das es für den Betrachter keinen Unterschied macht. Beispielsweise können <h1> und ein <div> gleich ausehen

Für den normalen Anwender der Seite ist die Semantik zumeist uninteressant. Jedoch hilft die Semantik bei bei drei großen Gruppen:

  1. Suchmaschinen

  2. Screenreadern

  3. Entwicklern

Layoutbereiche wurden vor HTML5 traditionell mit <div> erstellt, auch heutzutage ist das nicht falsch jedoch ergibt sich daraus ein Problem. Denn die <div> elemente sind semantisch neutral und haben keinerlei Bedeutung.

Durch die semantischen Elemente ergeben sich neue Möglichkeiten.

  • Eine Suchmaschine könnte mit den neuen Elementen z.B Suchbegriffe im Hauptinhaltsbereich main höher bewerten als in den anderen Bereichen wie nav und footer

  • Ein Screenreader könnte eine Funktion zum lesen des Hauptinhalts anbieten, mit dem der Benutzer direkt zum Inhaltsbereich oder zur Navigation springen könnte

  • Ein Browser könnte eine Funktion zum Lesen des Hauptinhalts anbieten. Auf Knopfdruck würden dann alle anderen Breiche ausgeblendet werden.

ARIA ist die Abkürzung für Accessible Rich Internet Applications und Teil einer Technischen Spezifikation der Web Accessibility Initiative(WAI), die sich um die Zugänglichkeit des Webs sorgen.

Dazu werden die HTML-Elementen um das Attribute role= erweitert, das einen fest definierten Wert erhält.

Time für Menschen und Maschinen

Das time-Element versucht, der Komplexität von Datumsangaben gerecht zu werden, indem es das Datum in zwei Teilen erfasst:

  • Die Zeichen zwischen <time> und </time> erscheinen im Browser. Hier kann man alles eintragen muss nicht mal ein Datum sein.

  • Das Attribut datetime enthält eine in der Spezification definiertes maschinenlesbare Variante. In der Reihenfolge Jahr(4 stellen), Monat und Tag mit jeweils 2 stellen.

    <time datetime="2015-04-01">1ster April 2015</time>

Uhrzeit mit Time

Eine Uhrzeit wird im Attribute datetime immer im 24-Stunden-Format notiert.

Wenn Datum und Uhrzeit zusammen angegeben werden, wird die Uhrzeit mit einem T oder mit einer Leerstelle vom Datum getrennt.

<time datetime="2015-04-01T13:00">1ster April 2015</time>
<time datetime="2015-04-01 13:00">1ster April 2015</time>

Es ist auch möglich sich auf die Zeitzone auf die man sich bezieht mit anzugeben. Dazu gibt es verschiedene Möglichkeiten:

Der Buchstabe Z hinter der Uhrzeit. Was zuviel bedeutet wie UTC/GMT Zeit für alle andern Zeitzonen wird ein + oder - verwendet. Dabei werden die von GMT abweichenden Stunden jeweils mit 2 Stellen notiert.

<time datetime="2015-04-01T13:00Z">1ster April 2015</time>
<time datetime="2015-04-01T13:00+01:00">1ster April 2015</time>

Hochauflösende Display CSS-Pixel vs Geräte-Pixel

Hochauflösende Bildschirme haben eine so hohe Pixeldichte, das das menschliche Auge die einzelnen Bildpunkte nicht mehr unterscheiden kann.

Auf Traditionellen Bildschirmen ist ein Pixel ein Pixel. Wenn man in CSS eine Breite von 320px definiert hat(css-pixel), wurden auf dem Bildschirm auch tatsächlich 320px zur Darstellung genutzt(Geräte-pixel). Dies änderte sich jedoch mit immer höher auflösenden Bildschirmen.

Wenn ein CSS-Pixel nicht mehr einem Geräte-pixel entspricht vom device-pixel-ratio bei einem Ratio 2:1 benötigt man doppelte Anzahl Pixel für ein Bilde um die Grapfik klar und scharf darstellen zu können.

Wenn also bei einem Ratio 2:1 eine Graphik mit 400x300 angezeigt werden soll benötigt man ein Bild mit 800x600

Online Ressourcen für Pixeldichte:

  • devicepixelratio.com, kann mit einem mobilen Device direkt angesurft werden.

  • mydevice.io, kann mit einem mobilen Device direkt angesurft werden.

  • screensiz.es, lange Liste von Devices

HTML und Responsive Graphiken

bilder sind auf responsiven Webseiten eine ganz bensondere Herausforderung, denn Grafiken sind im Web für einen Großteil der übertragene Daten verantwortlich, und es gab bisher noch keine HTML-Lösung für responsive Grafiken.

In CSS gibt es Media Queries, mit dennen man je nach Browserumgebung unterschiedliche Layouts und Hintergrundbilder ausliefen kann. Für das <img> Tag in HTML eingebundene Graphiken hingegen gab es eine solche Möglichkeit bislang nicht, sodass grafikintensive responsive Webseiten oft eher langsam waren, weil sie für alle Browser dieselben großen Grafiken im Gepäck hatten.

Vier Anwendungsfälle für responsive Grafiken.

  1. Unterschiedliche Pixeldichte
    Für normale und hochauflösende Bildschirme sollen jeweils passende Grafiken ausgeliefert werden. Auf nicht responsive Webseiten ist dies nicht das größte Problem.

    Damit für Bildschirme mit unterschiedlcher pixeldichte ausgeliefert werden können wird das img-tag verwendet.

    BSP: <img src="small.jpg" srcset="small.jpg x1 medium.jpg x2 large.jpg x3"

    Hierbei wird immer nur das gerade benötigte Bild heruntergeladen.

    Bei nicht definierten ratios wählt der Browser das nächst größere Element

    Die Viewportbreite spielt für diesen Anwendungsfall keine Rolle, es geht nur ums CSS-pixel zum Geräte-pixel ratio.

    Die bessere Alternative ist jedoch die Verwendung von der Attribute sizes und srcset mit w(es dürfen nur w oder x in einem Tag verwendet werden), weil mit dieser Kombination die Pixeldichte bereits berücksichtigt wird. bei doppelter pixeldichte veranschlagt er einfach doppelt soviele pixel für das Bild. Ein Viewport(vw) mit 320px benötigt also eine Grafik mit 640px.

  2. Unterschiedliche Viewport-Breite
    Für unterschiedlich breite Viewports werden in HTML jeweils passende Grafiken definiert, ähnlich wie in CSS mit Media Queries für Hintergrundbilder

    Um einem Brwoser zu sagen, dass er je nach Viewport-Breite eine bestimmte Graphik verwenden soll wird dem srcset Attribute hinter dem image source ein w angegeben. Das w signalisiert wie breit das Bild(pixelbreite) ist

    BSP: <img src="small.jpg" srcset="small.jpg 400w medium.jpg 800w large.jpg 1600w"

    Bis zu einer Breite von 400px erscheint das small.jpg, von einer breite von 401 bis 800px erscheint das medium.jpg und schlussendlich Ab 801px erscheint das large.jpg

    Mit dem Attribute sizes und vw die Anzeigebreite des Bildes bestimmen.

    Der Wert von sizes bildet die Grundlage für die Berechnung der geeigneten Grafik. vw ist die Maßeinheit viewport-width 100vw bedeutet, das eine Graphik den kompletten Viewport ausfüllen soll.

    Als Faust geregel gilt:

    • Wenn im Attribut scrset ein w-Wert auftaucht, dann sollte auch ein Attribut sizes vorhanden sein.

    • Falls keins vorhanden ist, fügt der Browser den Standardwert sizes="100vw" ein.

    BSP: <img src="small.jpg" srcset="small.jpg 400w medium.jpg 800w large.jpg 1600w" sizes="100vw"

    Das Attribute sizes kann die Viewport-width abfragen

    BSP: <img src="small.jpg" srcset="small.jpg 400w medium.jpg 800w large.jpg 1600w" sizes="(min-width:600px) 50vw, 100vw"

    Der Beispielcode bedeutet:

    • Ist der Viewport mindestens 600px breit (min-width:600px), dann soll das Bild halb so breit wie der Viewport dargestellt werden(50vw)

    • Ist der Viewport kleiner, soll es den gesamten Viewport einnehmen(vw100).

  3. Unterschiedeliche Bildmotive(Layout)
    Je nach Browserumgebungen sollen unterschiedliche Bildausschnitte oder Motive gezeigt werden.

    Benötigt das picture tag und den source Tag.

    Im gegensatz zu den srcset und sizes Attribute alles sich mit dieser variante auch verschiedene Bilder übermitteln.(Ja Diggi, gehacked wäre das auch mit dem anderen Attributen gegangen.)

    Wichtig dabei ist zubeachten, das der Browser hierbei von oben nach unten vorgeht. Das erste Bild was passt wird ausgewählt.

                    <picture>
                    <source media="" srcset="" sizes></source>
                    <source media="" srcset="" sizes></source>
                    <img src="" alt="">
                </picture>

    Das <picture> Element hat keine eigenen Attribute und legt sich lediglich als Wrapper um die Elemente

    Das <source> Element erledigt die Auswahl der Graphiken es kann diese Attribute besitzen:

    • srcset(erforderlich) Stellt eine Liste der Kandidaten zur Verfügung, für die am besten geeignetes Bild.

    • sizes(optional) Dient zur Beschreibung der gewünschten Darstellungsbreite.

    • media(optional) Enthält eine gültige Media Query, wie in CSS

    Der klassische Workflow des Webdesigns kennt die drei Stufen Gestalten, Nachbauen und Anschauen

    BSP Verschiedene Bilder je nach Breite des Viewports: 

                    <picture>
                    <source media="(min-width:600px)" srcset="workflow-waagerecht.png 1x, workflow-waagerecht.png 2x">
                    <source srcset="workflow-senkrecht.png 1x, workflow-senkrecht-large.png 2x">
                    <img src="workflow-fallback.png" alt="Klassischer Workflow">
                </picture>

  4. Unterschiedliche Dateiformate
    Es soll möglich sein, verschiedene Browsern unterschiedliche Dateiformate zu schicken.

    Hier für bekommt das source Element ein Attribut type womit der Mime-Typ der Datei angeben wird

    <source media="(min-width:600px)" srcset="workflow-waagerecht.webp 1x, workflow-waagerecht.webp 2x" type="image/webp">

<picture> vs srcset w

Bei img mit srcset w gibt der Auter de Webseite dem Browser,lediglich Hinweise, die Entscheidung wird aber im end effekt dem Browser überlassen. denn dieser hat alle benötigten informationen um das Richtig Bild auszuwählen. Bei picture hingegen hat der Browser keinen Spielraum durch die mediaqueries wird dem Browser genau vorgeschrieben welches Bild er verwenden soll.

Für bestimmte Art Direction und bestimmte Graphik Formate ist das in Ordnung jedoch sollte im allgemeinem die srcset w img Variante bevorzugt werden.

HTML5 und Sound

In HTML5 ermöglichen es die Elemente audio und video es den Browsern, Audio und Videodateien direkt und ohne Plugin abzuspielen. Leider sind sich die Browserhersteller nicht einige welche Formate hierfür verwendet werden sollen.

Um einen möglichst großen Benutzerkreis abdecken zukönnen sollten folgende Formate für den jeweiligen Browser vorliegen:

  • MP3 für Chrome, Safari, IE9+, iOS und Android 2.3+

  • OGG Vorbis für Firefox, Chrome und Opera

BSP:

        <audio id="audio" controls preload="metadata">
            <source src="url-to-audio.mp3" type="audio/mp3"></source>
            <source src="url-to-audio.ogg" type="audio/ogg"></source>
            <a href="url-to-audio.mp3">Download MP3 Audio</a>
        </audio>

Erklärung

Das AudioTag wird vom Browser von oben nach unten abgearbeitet. Die erste Datei die er interpretieren kann wird abgespielt alle andern werden ignoriert.
Daher sollten die Dateien IMMER in der Reihenfolge 1.MP3 2.OGG 3.Download-link vorliegen.

Der audio Tag besitzt 2 Attribute:

  • controls: lädt die Bedienelemente für den Browser eigenen Player.

  • preload: Bestimmt was der Browser laden soll, BEVOR der Benutzer auf den Play Button drückt. Es gibt foldenden Werte für das Attribute

    • metadata: lädt metainformationen herunter wie die songlänge, jedoch nicht den Song selbst

    • none: es wird nicht im vorraus geladen

    • auto: hierbei entscheidet der Browser was geladen wird. Die meisten beginnen mit dem Download der Datei.

Das source Tag besitzt 2 Attribute:

  • src: gibt den Pfad zur Datei an.

  • type: definiert mime-type der Audiodatei.

HTML5 und Video

BSP:

        <video id="video" controls poster="bild.jpg" preload="metadata">
            <source src="url-to-video.mp4" type="video/mp4"></source>
            <source src="url-to-video.webm" type="video/webm"></source>
            <a href="url-to-video.mp4">Download video</a>
        </video>

Für Safri muss das erste source ein mp4 file sein sonst werden alle anderen ignoriert.
